# Internationalization (next-intl)

## Locales

| Code | Language        | Notes                                      |
| ---- | --------------- | ------------------------------------------ |
| `en` | English         | Default; full strings in `messages/en.json` |
| `de` | German          | Full strings in `messages/de.json`        |
| `tw` | Twi / Akan      | Partial `messages/tw.json`, merged on top of English (fallback for missing keys) |
| `wo` | Wolof           | `messages/wo.json` merged over English + `messages/en/privacy.json` — native speaker review recommended for consent/ethics wording |

**Pattern banner (`landscape.pattern_banner`)** — Describes **session-local** statistics (trait combinations counted only among responses recorded in the same browser tab/process unless you add server-side aggregation). Keep wording honest when translating.

Routing uses `next-intl` with `localePrefix: 'as-needed'`: English URLs omit the `/en` prefix (e.g. `/questionnaire`); `de`, `tw`, and `wo` use `/de/...`, `/tw/...`, and `/wo/...`.

## Browser and saved language

- **Accept-Language** and the **next-intl locale cookie** are handled by the library middleware (see `src/middleware.ts`).
- **`localStorage`** key `pcms-locale-preference` stores the user’s explicit choice from the language switcher. On load, `LocaleFromStorage` may call `router.replace` with that locale so the client preference can override a first paint that did not yet match storage.

## Wolof: native speaker review

All Wolof strings in `messages/wo.json` should be reviewed by a **first-language Wolof speaker** for naturalness, spelling (Latin orthography choices), and fit for research/consent contexts—especially `consent.*`, `ethics_*`, and legal-adjacent copy.

## Twi: native speaker review (`[NEEDS_REVIEW]`)

All Twi strings are **draft** for research UX. Any message that still contains the literal tag **`[NEEDS_REVIEW]`** (or appears in the list below) should be checked by a **Twi/Akan first-language speaker** for naturalness, orthography (e.g. Akan spelling conventions), and cultural fit.

### High priority (user-facing, philosophical, or legally sensitive)

- **Ethics / consent / results copy** — Any new keys under `ethics_*`, `consent.*`, `results.assent*`, or delete-data UI in `messages/tw.json` should be reviewed the same way as consent strings below.
- **`welcome.title`** — Intended gloss: “Know thyself” (`Wo nim wo ho`); confirm register and common written form. **Avoid classical Greek framing in English UI for young audiences** if you fork strings; plain “understand yourself” is often clearer across cultures.
- **`welcome.subtitle`** — Full sentence marked in JSON.
- **`consent.title`** — Research consent framing.
- **`consent.agree`** — “Me de aba” (I agree); confirm for informed-consent context.
- **`questionnaire.question`** — “Asemmisa”; confirm as the label for “Question” in this UI.
- **`results.title`** — “Wo nkomo mu nsem” (results / what came out of the encounter); confirm against intended meaning of “results”.

### Dimensions (`dims.F` … `dims.V`)

For each dimension, review **`title`**, **`short_label`**, **`low_label`**, **`high_label`**, and **`description`**.

- **`dims.E` (Social energy)** — Framed with **Ubuntu** (“I am because we are”) and Ghanaian communal life; confirm that this matches how participants should read the **Western introvert/extravert energy** contrast without mis-translation.
- **`dims.R` (Structure preference)** — Connects **Ghana time**, communal scheduling, and flexibility vs clock-bound planning; confirm wording and that it still matches the underlying psychometric contrast.

### Other bracketed inline notes

Scan `messages/tw.json` for any remaining **`[NEEDS_REVIEW: …]`** comments; each describes the validation needed.

## Question item text (stems) vs UI language

- **Adaptive item wording** is stored in `content/questions/` (classic: `universal/core.json` + `refinement.json`; optional `global-behavioral-v2` when `NEXT_PUBLIC_PCMS_QUESTION_SOURCE=global_v2`). The canonical language of those JSON files is **English**.
- **German stems** for the classic universal bank are applied at **display time** via `src/i18n/question-stems-de.ts` (`QUESTION_STEMS_DE`, keyed by question `id`) and `resolveQuestionDisplayText()` — scoring and stored history still reference the same `questionId` and English bank for reproducibility.
- **`tw` / `wo`**: item stems remain English until optional overrides are added the same way (or the bank is duplicated in another language).
- **`global_v2` (200 items)**: no bundled non-English stems yet; extend `QUESTION_STEMS_DE` (or a locale-specific map) with the same `id` values as in `global-behavioral-v2/bank.json` when translations exist.

## Developer notes

- UI copy lives under **`messages/`** (`en.json`, `de.json`, `tw.json`, `wo.json`). The legacy `content/ui/*.json` tree is **not** used by the app.
- Runtime messages are loaded in `src/i18n/request.ts`. English loads **`messages/en.json`** plus **`messages/en/privacy.json`** (namespace `privacyPage`). German loads **`messages/de.json`** plus **`messages/de/privacy.json`**. Twi merges **`messages/tw.json`** over that English bundle via `deepMergeMessages`. Wolof merges **`messages/wo.json`** over the English bundle (fallback English + privacy where Wolof has no override).
- Legal/privacy strings for DE/EN live primarily in **`messages/de/privacy.json`** and **`messages/en/privacy.json`**; update those when deployment facts change (hosting, Supabase region).
- **`tw`** and **`wo`** override **`ethics_consent.storage_notice`**, **`privacy_body`**, **`rights_body`**, and **`site_footer`** where translated — full Datenschutz copy remains English via merge unless you add more keys or a locale-specific privacy merge.
- Client components that still expect a **flat** `Record<string, string>` (charts, pipeline copy) use **`useUiStrings()`** (`src/lib/use-ui-strings.ts`), which flattens nested JSON keys with dot notation (e.g. `results.loading`).
- Navigation and links that must stay locale-aware should import from **`@/i18n/navigation`**, not `next/link` / `next/navigation` directly.

## Adding a locale and avoiding hardcoded copy

1. Register the locale in **`src/i18n/routing.ts`** (and any static export lists that enumerate locales).
2. Load messages in **`src/i18n/request.ts`**: either a full JSON file (like `de`) or **`deepMergeMessages(en, …)`** over English (like `tw` / `wo`) so missing keys fall back cleanly.
3. Add **`messages/<locale>.json`** (and **`messages/<locale>/privacy.json`** if you need a full legal bundle like DE/EN). Keep **nested keys in sync** with `messages/en.json` for anything you translate; partial locales rely on merge fallback for the rest.
4. In **`src/app/[locale]/…`** and shared **`components/`**, avoid **literal user-facing English**: use `useTranslations` / `next-intl`, or **`useUiStrings()`** + **`formatUiString()`** for flat keys and placeholders. Internal-only or research tooling may stay English by convention.
